# [ADR-0012] Schema-Validated Dynamic RPC Pattern

## Status
Proposed (2026-04-29)

## Context
DXOSでは開発効率と柔軟性の向上のため、`useMasterCRUD.ts` 等を通じて動的なテーブル更新を行う汎用的な RPC（`rpc_execute_master_update`）を使用している。しかし、単純な動的SQLはSQLインジェクションや不正なカラムへのアクセス、意図しないテーブルの改ざんといったリスクを伴う。

## Decision
今後、すべての動的SQLを伴う RPC は以下の「三段構えのバリデーション」を物理層（PL/pgSQL）において実装することを義務付ける。

### 1. テーブル・ホワイトリスト (Authorization)
更新を許可するマスタテーブル名をハードコードまたはメタデータテーブルで定義し、外部から渡されたテーブル名がリストに含まれているか厳密に照合する。
- 違反時のエラーコード: `DXOS_VAL_01`

### 2. スキーマ・イントロスペクション (Integrity)
`information_schema.columns` を参照し、更新対象として渡された JSON のキーが実際にターゲットテーブルの物理カラムとして存在するかを検証する。
- 違反時のエラーコード: `DXOS_VAL_02`

### 3. エラーコード標準 (Messaging)
例外発生時はフロントエンドがパース可能な `DXOS_VAL_` プレフィックス付きのメッセージを送出する。

## Consequences
- **Positive**: セキュリティリスクの根本的排除。DBスキーマとコードの不整合が実行時に即座に判明する。
- **Negative**: RPC実行時にわずかなオーバーヘッド（スキーマ照合クエリ）が発生する。新規テーブル追加時にホワイトリストの更新が必要になる。
